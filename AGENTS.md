## Running Playwright E2E locally

`npm run test:e2e` boots the app via `server.js` (`playwright.config.ts` webServer runs `npm run build && npm run start`), which requires `NODE_ENV=production`. `server.js`'s `@bitc/cnad` calls (a cPanel-only auto-deploy/restart helper) are guarded behind `if (process.env.NODE_DIR)` as part of the Azure Container Apps migration (see the Deployment section below), so leaving `NODE_DIR` unset - the default in this sandbox and in the container image - skips `cnad` entirely and no workaround is needed to get the server to bind a port.

In a minimal sandbox with no system Chromium deps and no root access, `apt-get download <pkg>` (no sudo needed) + `dpkg-deb -x <pkg>.deb <prefix>` + `LD_LIBRARY_PATH=<prefix>/usr/lib/x86_64-linux-gnu` is a viable way to satisfy Chromium's shared-library requirements (libnspr4, libnss3, libatk*, libgtk-3-0, libpixman-1-0, libharfbuzz0b, libgraphite2-3, etc.) without `playwright install --with-deps`, which needs a password-less sudo that may not be available.

## Test hooks in components

`ElegantContactForm` (and its Playwright spec in `e2e/contact-form.spec.ts`) use `data-testid` attributes for stable selectors. Radix Themes components (`TextField.Root`, `TextArea`, `Select.Trigger`, `Button`) forward arbitrary DOM props, including `data-testid`, straight through to the real underlying `<input>`/`<textarea>`/`<button>` element (verified by reading `node_modules/@radix-ui/themes/dist/cjs/components/*.js`), so adding `data-testid` to these components works exactly like adding it to a plain HTML element.

`PortfolioPageContent` renders its children twice (a desktop wrapper and a mobile wrapper, toggled by CSS), so every `data-testid` in `ElegantContactForm` exists twice in the DOM at once. Any Playwright selector built on these testids needs `.first()` to land on the visible desktop copy under the default viewport.

## The Live System hero (career topology dashboard)

`app/sections/LandingSection.tsx` (home) and `app/components/sections/GenericLandingSection.tsx` (`/experience`, `/projects`) both render the shared `SystemsHero` component (`app/components/SystemsHero.tsx`), which wraps `TopologyDashboard` (`app/components/TopologyDashboard.tsx`) - an SVG career-topology diagram (his five employers as nodes orbiting a center "SHAGAN" node, with `getPointAtLength` + `requestAnimationFrame`-animated packets along curved SVG paths) plus a stat strip. This replaced the old identical "Hi, I'm Shagan, Full Stack Engineer" hero duplicated across both files.

The node graph is data-driven: `app/lib/topology.ts`'s `buildTopologyNodes()` maps the same `WpProjectApiResponse[]` experience data `ExperienceSection.tsx` already fetches (filtered to `WORDPRESS_CATEGORIES.WORK_EXPERIENCE`) into `TopologyNode[]`. When `fetchAllWpProjects()` returns `false`/empty (no `WP_DOMAIN`, or a WP outage - the same failure mode `ExperienceSection`/`ProjectsSection` silently return `null` for), it falls back to `FALLBACK_TOPOLOGY_NODES`, a static seed of his five real employers (Broadway Media, BSure, Vodacom/VodaPay, NewNow, Aegle Digital) with year-precision approximate tenure dates, so the hero never renders empty in this network-less sandbox.

The stat strip deliberately avoids fabricated telemetry (no fake "P99 latency" jitter): all four stats - career uptime (ticking, computed from the earliest node's real `startDate`), employer count, active-since year, and current employer - are genuinely derived from the node data, not `Math.random()` jitter dressed up as metrics.

`TopologyDashboard` uses two distinct layouts, not one CSS-scaled viewBox: desktop (`DESKTOP_VIEWBOX`, 1120x420) spreads nodes on an ellipse with side-anchored labels, since a `<main>` `flex-row` ancestor's default flex-item sizing is wide enough there; mobile (`MOBILE_VIEWBOX`, 420x820) stacks nodes vertically with centered labels below each node, because there's no horizontal room on a narrow viewport for long, data-driven company/role strings next to a left/right-spread node. Both layouts truncate label/sub text (`truncate()` helper) since real WP company/role strings can be arbitrarily long - this isn't just a fallback-data cosmetic fix, it's load-bearing for real data.

`SystemsHero`'s outer `<Section>` needs explicit `w-full min-w-0` classes: without `min-w-0` the flex item sizes to its text content's preferred width instead of shrinking to the viewport - on mobile this clipped the heading off-screen with no visible bug until measured (`overflow-hidden` on `main` hid it rather than showing a scrollbar). At the time this was added, `GenericContentPage` rendered it as a direct child of `MainLayoutClient`'s `<main className="flex ... flex-row ...">` with no intermediate `w-full` wrapper (unlike the homepage, where `PortfolioPageContent`'s own wrapper div already has `w-full`). The later scroll-clipping fix below gave `GenericContentPage` its own `w-full min-w-0` wrapper div too, but that div isn't itself a flex container (no `flex` class, just `flex-1` as a flex item of `main`), so `SystemsHero`'s `Section` is now a plain block child rather than a direct flex-row item - its own `w-full min-w-0` classes were left in place as-is and are harmless either way.

The homepage (`/`) had no hero at all before this change - `LandingSection.tsx` existed but was never imported by `app/page.tsx` (which renders `PortfolioPage` straight into `SummarySection`), a leftover from a past "Major UI update" refactor. `PortfolioPage.tsx` now renders `<LandingSection />` as the first child inside `PortfolioPageContent`, ahead of the `#summary` block, so the homepage - the link actually sent to hiring managers - carries the redesign too, not just the two secondary pages.

Reduced motion: `TopologyDashboard` uses `usePrefersReducedMotion()` (the same hook `MatrixRain.tsx` uses) to skip the packet `requestAnimationFrame` loop entirely and freeze packets at each edge's midpoint, and to drop the status-dot `animate-pulse` class - mirroring `MatrixRain`'s existing reduced-motion handling rather than inventing a new pattern.

`SystemsHero`'s "About Me" CTA (`app/components/SystemsHero.tsx`) links to `/#summary` - the homepage (`/`) has no dedicated About page, the "About" content is `SummarySection` (`app/sections/SummarySection.tsx`), rendered inside `PortfolioPage.tsx` wrapped in `<div id="summary">`. That id is the only stable anchor for "About Me" CTAs; `/#summary` (the leading slash forces a cross-page navigation before the hash is applied) works whether the CTA renders on the homepage itself or on `/experience`/`/projects`, which render through `GenericContentPage`/`GenericLandingSection` and don't have a `#summary` element of their own.

`app/experience/sections/ProjectsSection.tsx` (dead code, despite the name it exported `AboutSection`, never imported, had the same broken `/about` link plus a stale tech list - see the redesign scout's report Finding 6) was deleted as part of this same broken-link cleanup.

`MainLayoutClient.tsx`'s `<main className="flex flex-grow flex-row overflow-hidden w-full h-full">` hard-caps every route's content to exactly one viewport's height with no scrollbar, since `overflow-hidden` + `h-full` clips rather than scrolls. The homepage survives this because `PortfolioPageContent.tsx` wraps its own children in a `.portfolio-scroll-container` (`flex-1 overflow-y-auto`) that does the actual scrolling before content ever reaches `<main>`'s clipping boundary. `GenericContentPage.tsx` (used by `/experience` and `/projects`) had no equivalent wrapper - it rendered `<GenericLandingSection /><ExperienceSection />` as a bare fragment - so on those two routes real content (most of the topology hero's employer nodes and its entire stat strip on mobile) was silently clipped with nothing to scroll. Fixed by wrapping `GenericContentPage`'s children in a `w-full min-w-0 flex-1 overflow-y-auto overflow-x-hidden` div - the same `flex-1 overflow-y-auto overflow-x-hidden` scroll pattern `PortfolioPageContent` already uses, plus the `w-full min-w-0` pair `SystemsHero`'s own `Section` needed for the same flex-sizing reason (see above) - rather than touching `MainLayoutClient` itself - `main`'s `flex-row` + default `align-items: stretch` means this single wrapper div becomes the one flex item that gets stretched to `main`'s clipped height, and it's a plain block element internally so the hero and section content still stack normally in document flow; scrolling now happens inside that wrapper instead of being lost. Any future route added through `GenericContentPage` inherits this scroll wrapper automatically; a route that bypasses `GenericContentPage` and renders straight into `MainLayoutClient`'s `<main>` would need the same wrapper pattern repeated.

`PortfolioPageContent.tsx`'s two content wrappers (`px-8 md:px-12 py-8 max-w-4xl mx-auto` on desktop, `px-4 py-6 max-w-4xl mx-auto` on mobile) both need `mx-auto` alongside `max-w-4xl`: on wide desktop viewports (verified at 2560x1440) the `max-w-4xl` cap left the content block sized correctly but sitting flush-left against the sidebar nav, with a large unused strip of empty space on the far right edge, because the capped-width block had no centering rule and defaulted flush-left within its `flex-1` parent. `mx-auto` centers the capped block in the remaining space instead. `mx-auto` is a no-op on the mobile wrapper at narrow viewports, where content already spans the full container width, but is included for consistency and to guard against the same flush-left bug reappearing if the mobile breakpoint's effective width ever exceeds `max-w-4xl`.

## Deployment (cPanel to Azure Container Apps migration)

The app migrated off cPanel to Azure Container Apps (ACA). The target was originally GCP Cloud Run; the captain switched it to Azure after a follow-up investigation found the two tied on technical fit and cost for this workload, with career/market skill transferability breaking the tie toward Azure. The legacy `deploy-to-cpanel.yml` workflow (internal name `CNAD`, FTP-deployed builds to the cPanel host on push to `main`/`stg`) was deleted once its builds started failing on every push against the stale cPanel target, generating repeated failure notifications with no working deploy behind them; `deploy-azure-container-apps.yml` (triggers on push to `main`, plus `workflow_dispatch` for manual re-runs/rollback-redeploys) is now the sole CI/CD deployment path. Nothing gets cut over at the DNS level until it is explicitly repointed in a later phase, so the live site still resolves to the (now undeployed-to) cPanel host in the interim.

`server.js`'s `cnad.*` calls remain, still guarded behind `if (process.env.NODE_DIR)`: `@bitc/cnad`'s `cnad.start()`/`cnad.watch()` were the live cPanel host's own `npm install`/restart trigger, invoked by the now-deleted `deploy-to-cpanel.yml`'s "Creating restart file" step. Removing `@bitc/cnad` from `server.js`/`package.json` was intentionally left out of the workflow-decommission - it's inert in every environment that doesn't set `NODE_DIR` (the container image, this sandbox, and CI all leave it unset), so it carries no functional risk to ACA, and deciding whether cPanel is retired enough to also delete the runtime dependency is a separate call from removing the failing CI workflow. `pm2` was dropped (unused - no `ecosystem.config.js` or `pm2` invocation anywhere; the container platform supervises the process instead).

`Dockerfile` is a two-stage build, not the single `npm ci --omit=dev && npm run build` sequence the current app's `next.config.mjs`/`app/lib/fonts.ts` would suggest is enough: `next build` needs devDependencies (`typescript`, `eslint`, and critically `tailwindcss`, which `next/font/google`'s PostCSS pipeline requires even though Tailwind isn't otherwise imported at runtime). So the builder stage runs a full `npm ci`, and only the final runtime stage reinstalls with `npm ci --omit=dev`, keeping the shipped image free of dev dependencies. The runtime stage also drops to the unprivileged `node` user (`node:20-slim`'s built-in account) rather than running as root. Verified by running the equivalent steps by hand (no Docker available in this sandbox to build the image directly): full install + build, then delete `node_modules` and reinstall with `--omit=dev`, then `node server.js` and curl it - this must succeed after any change to build tooling or `next.config.mjs`.

`server.js` reads `APP_PORT`, not any env var ACA injects - ACA has no Cloud-Run-style `PORT` convention, it just proxies to whatever `ingress.target_port` is configured on the Container App. The Dockerfile sets `APP_PORT=8080` and `terraform/container_app.tf`'s `ingress.target_port` must keep matching that value, since server.js's own env var scheme wasn't changed to keep this phase a pure infra addition.

`terraform/` is an Azure Container Apps root module: Container Registry (Basic SKU), a Container Apps environment + `azurerm_container_app` with `min_replicas = 0` (true scale-to-zero), a dedicated user-assigned identity for the app's own image-pull/Key-Vault access (separate from a second user-assigned identity + one `azurerm_federated_identity_credential` per branch for GitHub Actions OIDC), RBAC-authorized Key Vault + secret for `RESEND_API_KEY` referenced via a Key-Vault-backed `secret` block rather than a plaintext env var, and an Azure DNS zone for `shaganplaatjies.co.za`. It has no backend configured; phase 1 (`enable_custom_domain = false`) has since been applied against a real Azure subscription - the successful `deploy-azure-container-apps.yml` `workflow_dispatch` run (see the Deployment section above) confirms the registry, Container App, and OIDC identities it creates are live. `subscription_id`, `tenant_id`, and `container_image` have no defaults on purpose (populated in a gitignored `terraform.tfvars`, copied from `terraform.tfvars.example`). `terraform validate` passes (verified with a scratch-downloaded Terraform 1.9.8 binary and `terraform init` against the real `azurerm`/`azuread` provider schemas, since no `terraform` binary or Azure credentials exist in this sandbox); phase 2 (`enable_custom_domain = true`) still needs registrar access before it can be applied.

`terraform/frontdoor.tf` fronts the Container App with Azure Front Door (Standard) rather than binding a custom domain directly on the Container App: `container_app.tf`'s ingress only exposes the default `*.azurecontainerapps.io` FQDN, and TLS/custom-domain termination for `shaganplaatjies.co.za` happens at Front Door instead. This resolves what used to be an open gap - a literal CNAME at the DNS zone apex isn't DNS-compliant (Azure DNS, like virtually all DNS servers, rejects a CNAME coexisting with the zone's own NS/SOA records at the apex), which the prior GCP module never had to deal with since Cloud Run's global load balancer exposed a plain, A-record-able static IP. `dns.tf`'s `azurerm_dns_a_record.app` uses Azure DNS's alias-record feature (`target_resource_id` pointing at the Front Door endpoint) to alias the apex directly at an Azure resource instead of naming it via CNAME - the standard Azure pattern for apex-aliasing to a PaaS frontend. See `terraform/README.md`'s "Apex domain: fronted by Azure Front Door" section for the full resource breakdown.

Every domain-dependent resource - `dns.tf`'s `azurerm_dns_zone`/apex `azurerm_dns_a_record`/`_dns_txt_record`, and all of `frontdoor.tf` - is gated behind `var.enable_custom_domain` (`variables.tf`, default `false`) via `count = var.enable_custom_domain ? 1 : 0`, because the captain doesn't currently have registrar access for `shaganplaatjies.co.za` (mid-transfer away from cPanel hosting). A first apply with the default `false` stands up only the core app infra (registry, Container App environment/app, identities, Key Vault) reachable via the Container App's own default FQDN; a second apply flips the flag to `true` once DNS delegation access is back, to layer in the DNS zone and Front Door custom domain. See `terraform/README.md`'s "Two-phase apply: enable_custom_domain" section.

`terraform/frontdoor.tf`'s `azurerm_cdn_frontdoor_origin_group.app` deliberately carries no `health_probe` block: Front Door health probes poll the origin on their own interval regardless of real traffic, which would keep the Container App's single replica alive around the clock and defeat `container_app.tf`'s `min_replicas = 0` scale-to-zero. Do not add one back without also reconsidering scale-to-zero.

## Strudel live-coding music (`app/lib/hooks/useStrudelCycles.ts`, `app/components/tidal-cycles/MusicProject.tsx`)

The `@strudel/*` packages (`core`, `mini`, `transpiler`, `webaudio`) ship no `.d.ts` files of their own (verified across every published tarball on npm), so `app/lib/types.d.ts` carries hand-written ambient `declare module` shims permanently - this isn't a placeholder to delete once "real types land," there are none to land.

Getting a Strudel pattern string (like `app/lib/strudel/first-song.str`, which is mini-notation, not plain JS) to actually produce sound needs three setup calls before `repl().evaluate(code)`, not just constructing the repl: `miniAllStrings()` (from `@strudel/mini`) registers the mini-notation parser as the string parser, so calls like `note("...")` interpret their string argument as a pattern instead of a literal; `evalScope(core, mini)` (from `@strudel/core`) installs `note`, `gain`, `slow`, etc. as globals, because `repl().evaluate()` runs pattern code through `new Function(...)`, which has no closure over module imports; and `registerSynthSounds()` (from `@strudel/webaudio`, re-exported from the `superdough` package) registers the built-in waveform synths (`triangle`, `sine`, etc.) into superdough's sound map. Skipping that last call is the actual reason a plausible-looking Strudel integration can run with zero thrown errors, report "playing," and still be silent: the scheduler starts fine and evaluates fine, but every note-trigger fails individually inside `getTrigger`'s try/catch with `"sound triangle not found! Is it loaded?"` - a `console.log`, not a `console.error`, so it doesn't surface as a page error and is easy to miss without reading the console output verbatim.

Verifying this actually works requires more than checking `AudioContext.state === "running"` and no console errors - both were true even while every note was silently failing. What proved real playback: reading the exact console log lines (not just filtering by error/warning type), and patching `BaseAudioContext.prototype.createOscillator`/`createGain`/`AudioParam.prototype.setValueAtTime` before clicking play to confirm real node counts climb while playing and freeze the instant `stop()` is called.

No system Chrome exists in this sandbox and `chrome-devtools-axi`'s default auto-launch fails (`Could not find Google Chrome executable for channel 'stable'`). The Playwright chromium already cached at `~/.cache/ms-playwright/chromium-1228/chrome-linux64/chrome` works as a substitute: launch it directly with `--headless=new --remote-debugging-port=<port> --no-sandbox --disable-gpu --user-data-dir=<scratch dir>`, then point `chrome-devtools-axi` at it via `CHROME_DEVTOOLS_AXI_BROWSER_URL=http://127.0.0.1:<port>` (use `CHROME_DEVTOOLS_AXI_SESSION=<unique-name>` too - this sandbox runs multiple concurrent agent sessions that each default to the same bridge port/session). Background processes started with a bare shell `&` do not survive past the end of that Bash tool call in this harness; use `run_in_background` (or the manually-launched Chrome, which is a separate detached process) instead, or the server/browser will be gone on the next command. Never export these `CHROME_DEVTOOLS_AXI_*` vars via `~/.bashrc` or any other shared/persistent config - that file is not scoped to a single agent session and silently breaks other concurrent sessions relying on their own port/session defaults; set them inline per-command (or `export` them only within the current shell) instead.

## Ops Console nav/chrome reskin (part 1 of the Ops Console homepage redesign)

This is the first of several incremental PRs toward the "Ops Console" direction chosen from `/home/duke/firstmate/data/homepage-composition-r7`'s three prototypes (report + screenshots + `concepts/ops-console/index.html` - note this reference material lives under firstmate's own data directory, not inside this repo). This part only reskins navigation and page chrome (nav rail + top status bar) into the Ops Console dark/hairline/monospace visual language; it deliberately does not touch `TopologyDashboard`/`SystemsHero` or build the two-panel dashboard (service inventory list, deploy-log ticker, full-bleed stat strip) the prototype also shows - those are separate follow-up parts.

**Chrome tokens (`public/styles/globals.css`):** `--ops-bg`, `--ops-line`, `--ops-line-bright`, `--ops-fg-0/1/2`, `--ops-accent` are defined at `:root` (light fallback, aliased to the existing `--gray-*`/`--cyan-*` Radix scale so the light theme keeps looking like the rest of the light-mode site) and overridden under `html.dark` with the prototype's literal near-black palette (`#050708` bg, `#1c262a` hairlines, `#4dd6c4` accent, etc.). **Do not write a bare `.dark { ... }` selector for these or any future ops-console tokens** - `@radix-ui/themes`' own `Theme` component renders wrapper divs carrying class `"radix-themes dark"` / `"radix-themes light"` purely for its own internal palette scoping, unrelated to this site's light/dark toggle (which only ever adds/removes `"dark"` on `document.documentElement`). Root layout's outermost `<Theme appearance="dark">` is permanently one such wrapper, so a bare `.dark` selector matches it too and, because custom properties inherit down through the nested per-page `<Theme appearance={appearance}>` (which doesn't reset arbitrary non-Radix custom properties, only its own named ones like `--gray-1`), makes the chrome render dark even while the toggle is set to light. Caught by comparing `getComputedStyle` of `--gray-1` (correctly toggled, `#fcfcfc` in light mode) against `--ops-bg` (incorrectly stuck dark) on the same element and walking `parentElement` to find the stray `DIV.radix-themes.dark` ancestor. Scope strictly to `html.dark`.

**Nav rail (`app/components/PortfolioNav.tsx`)** now runs in two modes off the same component rather than being duplicated: when `onSectionChange` is passed (homepage), it behaves exactly as before - anchor links with `preventDefault` + scroll-to-section, active state from the `activeSection` prop. When `onSectionChange` is omitted (`/experience`, `/projects`), it renders real `next/link` `Link`s instead: sections with a dedicated route (`experience`, `projects`, via a small `PAGE_ROUTES` map) link straight there; the rest (`summary`, `music`, `blog`, `contact` - homepage-only anchor sections) link to `/#id` (leading slash forces the cross-page navigation before the hash applies, same pattern `SystemsHero`'s "About Me" CTA already established), and active state comes from `usePathname()` instead of scroll tracking. The section list itself (`PORTFOLIO_NAV_SECTIONS` in `app/lib/constants.ts`) is now a single shared source of truth imported by both `PortfolioPage.tsx` (homepage) and `GenericContentPage.tsx` (`/experience`, `/projects`), replacing what used to be an inline array only `PortfolioPage.tsx` had.

**`/experience` and `/projects` previously rendered no nav at all** - the old `PortfolioNav` was only ever mounted by `PortfolioPageContent.tsx` (homepage-only); `GenericContentPage.tsx` rendered `<GenericLandingSection /><contentComponent />` as a bare fragment with nothing else (this was already flagged as Finding 3 in the `homepage-composition-r7` scout report). Since this task's "apply consistently across all three pages" requirement can't be satisfied by restyling a nav that doesn't exist on two of the three pages, `GenericContentPage.tsx` now also renders `PortfolioNav` (link mode) in the same nav-rail-plus-scrollable-content-pane flex split `PortfolioPageContent` already uses, reusing the same `w-full min-w-0 flex-1 overflow-y-auto overflow-x-hidden` scroll wrapper the earlier clipping fix (see "Live System hero" section above) established - this is a real, deliberate scope expansion beyond a literal reskin, not an oversight.

**Top status bar (`app/components/StatusBar.tsx`)** replaces `Header.tsx` (deleted, along with its sole consumer `AnimatedIconStack.tsx`, which becomes fully orphaned once `Header.tsx` is gone - `IconsList.tsx`/`icons/NavIcons.tsx`/`components/Footer.tsx` were left alone since they have other, pre-existing, unrelated consumers or were already dead before this change and auditing them is out of scope here) in `MainLayoutClient.tsx`, in the exact same layout slot (full-width, above `<main>`, spanning both the nav rail and content pane below it). This is a deliberate deviation from the Ops Console prototype, where the topbar only spans the content pane to the right of the rail (the rail's own brand header sits above it on the left). Moving the topbar into that per-page position would require hoisting nav rendering out of `PortfolioPageContent`/`GenericContentPage` and into the shared layout, which starts entangling the homepage's scroll-tracking nav mode with the shared chrome - treated as out of scope for a "nav/chrome reskin only" PR. A full-width top bar above everything (rail included) is itself a common real ops-dashboard pattern (Grafana, Vercel, Datadog all do this), so this reads as a reasonable, low-risk simplification rather than a visual compromise; revisit if a later part in this sequence wants pixel parity with the prototype.

**Pre-existing latent bug surfaced (and fixed) by making the nav functional on mobile for the first time:** `PortfolioPageContent.tsx` renders `{children}` twice - once inside a `hidden sm:flex` desktop wrapper, once inside an `sm:hidden` mobile wrapper (see the existing `data-testid`-duplication note above) - so every section `id` (`#summary`, `#experience`, etc.) exists twice in the DOM simultaneously. The old code's `document.getElementById(sectionId)` and `document.querySelector('.portfolio-scroll-container')` both always resolved to the first (desktop) copy regardless of viewport, which was silently harmless before because the mobile nav's section list (`<ul>`) was `hidden sm:block` - i.e. never rendered on mobile at all, so `handleSectionChange` was simply unreachable on a phone. This Ops Console reskin makes the compact dot-row section list visible and clickable on mobile (closer to the prototype's mobile rail, and a functional improvement over the prior "no mobile section nav at all"), which immediately exposed the bug: clicking a section on mobile scrolled the invisible desktop copy, doing nothing visible. Fixed with a `getVisibleElementById` helper (`Array.from(document.querySelectorAll(...)).find(el => el.offsetParent !== null)`) used by both the click handler and the scroll-spy's `IntersectionObserver`-less scroll listener (which is now attached to *both* `.portfolio-scroll-container` instances, not just the first).
